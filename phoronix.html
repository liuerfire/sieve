Linux 7.0-rc4 修复大型系统上的挂起与性能回退问题

Sun, 15 Mar 2026 06:52:00 -0400 Open original
Recommended

标题聚焦Linux 7.0调度器性能修复，属Linux桌面技术与内核优化，符合Source interest（Linux桌面技术）及Global interest（开源项目、软件新功能），但非Rust或新发行版等高优先级主题，故为recommended。


本周 Linux 7.0 合并窗口发布了两个紧急修复补丁集，分别解决调度器（scheduler）和 x86 架构中的严重问题。

首先，Thomas Gleixner 主导修复了自去年 11 月 mm/cid 重写后引入的多个调度器问题。这些问题包括：并发 fork 导致的 CID 挂起、vfork()/CLONE_VM 场景下的 MMCID 错误引发的挂起、以及在大型系统上因使用 for_each_process_thread() 遍历任务列表而导致的性能回退。新方案采用更高效的 sched_mm_cid::node 链表替代原有计数逻辑，显著提升性能。

其次，x86/urgent 补丁集修复了一个与挂起到内存（suspend-to-RAM, s2ram）相关的硬件兼容性问题。当固件在恢复时意外重新启用 x2apic 模式（而内核在启动时已禁用它），会导致内核继续使用 xAPIC 接口但硬件处于 x2APIC 模式，从而引发系统挂起。新补丁确保在 lapic_resume() 中，若内核预期 x2apic 被禁用（x2apic_mode = 0），则会强制将其关闭，避免此问题。

这些修复预计将在今日合并，并随 Linux 7.0-rc4 版本发布。

开源 GreenBoost 驱动通过系统内存与 NVMe 扩展 NVIDIA GPU 显存以运行更大 LLM

Sat, 14 Mar 2026 20:17:00 -0400 Open original
Recommended

标题描述开源GreenBoost驱动增强NVIDIA GPU内存能力以支持大模型，涉及AI模型应用（Global high interest）与开源驱动开发（Global interest），但NVIDIA属非KDE/Firefox/Chrome等Source high interest项，且硬件驱动偏底层，故为recommended而非critical。


独立开发者 Ferran Duarri 发布了一个名为 GreenBoost 的开源 Linux 内核模块（GPLv2 许可），旨在通过系统 RAM 和 NVMe 存储扩展 NVIDIA 独立 GPU 的专用显存（vRAM），从而更轻松地运行大型语言模型（LLM）。该方案不替换 NVIDIA 官方驱动，而是作为补充：包含一个内核模块（greenboost.ko）和一个用户态 CUDA shim 库（libgreenboost_cuda.so），后者通过 LD_PRELOAD 注入，透明地拦截 CUDA 内存分配调用。小内存请求直接交由原生 CUDA 处理，而大块内存（如超出显存的模型权重或 KV 缓存）则被重定向至内核模块分配的固定系统内存页，并通过 DMA-BUF 导出为 CUDA 可识别的设备内存。PCIe 4.0 x16 链路负责数据传输（约 32 GB/s），并通过 sysfs 接口实时监控使用情况。此外，shim 还巧妙地拦截了 dlsym 调用，以确保像 Ollama 这类通过动态符号解析获取 GPU 信息的程序能正确识别扩展后的内存总量，避免因仅看到物理显存容量而将计算层卸载到 CPU。项目代码已托管于 GitLab。

Linux 内核 API 规范框架结束 RFC 阶段，进入正式提交流程

Fri, 13 Mar 2026 12:18:47 -0400 Open original
Critical

Linux内核API规范框架推进，属Linux内核基础设施建设，契合Global high interest（编程工具发布/分享、开源项目）及Source interest（Linux桌面技术依赖稳定内核API），且为开发者工具类，故为critical。

经过五轮带有“征求意见（RFC）”标记的评审后，Linux 内核 API 规范框架（Kernel API Specification Framework）的最新补丁系列已移除 RFC 标记并正式提交。该框架旨在以人类可读和机器可解析的形式，更好地记录内核 API。

据补丁系列的作者 Sasha Levin 介绍，此提案引入了一套机制来文档化内核 API，以解决长期以来内核与用户空间程序之间接口稳定性维护的难题。尽管内核承诺永不破坏用户空间，但由于缺乏机器可读的 API 规范，系统调用（system calls）和 IOCTL 接口仍时有损坏。新的规范可以详细记录参数类型、有效范围、约束条件和对齐要求，并明确返回值语义（包括成功条件和错误代码的含义）。此外，执行上下文要求、所需能力（capabilities）、锁约束、信号处理行为及副作用等均可被形式化地指定。

这些规范与所描述的代码共存，既便于人类阅读，也能被机器解析。在启用 CONFIG_KAPI_RUNTIME_CHECKS 配置选项时，可在运行时进行验证；也可通过 debugfs 导出供用户空间工具使用；还能从 vmlinux 或源代码中提取。这为静态分析工具在编译时验证用户空间 API 使用、基于形式化规范生成测试、一致性错误处理验证、自动化文档生成以及内核接口的形式化验证提供了可能。

该实现包含一个核心框架，用于在 ELF 节中存储规范数据，并集成了 kerneldoc 以支持内联规范书写，同时提供了 debugfs 接口用于运行时查询。项目还包含一个基于 Rust 的提取工具（tools/kapi），支持输出 JSON、RST 和纯文本格式。目前，已为四个基础文件系统调用（sys_open、sys_close、sys_read、sys_write）提供了示例规范。此次提交还包括一个包含 38 项测试的 KUnit 测试套件和一个包含 29 项以上 TAP 测试的运行时验证自测程序。

本次更新的补丁除了移除 RFC 标记外，还基于 Linux 7.0-rc1 进行了重定基（rebase），并改进了 KUnit 测试覆盖率、简化了示例规范，并完善了运行时验证自测。
